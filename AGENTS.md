# AGENTS.md

本文件给所有参与本项目的 AI Agent、自动化编码助手和开发者阅读。  
开始任何修改前，先读完本文，再阅读当前任务相关代码。

## 项目中心思想

这是一个由 AI LLM 自主维护和运营的网站系统。项目目标不是做一个需要人工频繁编辑的 CMS，而是让网站在极少人工干预下持续完成选题、采集、写作、审核、发布、SEO、数据分析和自我改进。

所有升级都必须服务这个中心思想：

- 优先提升自治能力、内容质量、稳定运行和自我恢复能力。
- 不要引入需要人工长期维护的流程。
- 不要把问题转嫁给人工审核、人工选题、人工编辑。
- 程序可以接受较长等待、较多重试和后台慢任务，只要能提高持续运行成功率。
- AI 生成内容必须尽量降低模板味、报告味和明显 AI 味。

## 技术栈与结构

项目是 Node.js + Express + EJS 应用。

主要目录：

- `server.js`：应用入口、路由挂载、启动调度器。
- `routes/`：前台、后台和 API 路由。
- `views/`：EJS 模板，`views/pages` 是前台页面，`views/admin` 是后台页面。
- `public/`：前端 CSS、JS、favicon 等静态资源。
- `ai/`：AI Agent、Prompt、搜索、写作、质量检测、Provider 客户端。
- `scheduler/`：定时任务、冷启动、狂暴模式、文章生成调度。
- `db/`：JSON 文件数据库封装和初始化逻辑。
- `tests/`：Node 内置测试套件。

数据主要保存在 `data/db.json`。线上数据比本地重要得多；需要样本时优先只读查看线上数据，不要假设本地数据完整。

## Git 与上线流程

每个功能或修复都必须使用独立分支，分支名默认使用 `codex/` 前缀。

推荐流程：

```bash
git switch master
git pull --ff-only origin master
git switch -c codex/<task-name>

# 修改、测试、提交后
git switch master
git pull --ff-only origin master
git merge --no-ff codex/<task-name> -m "merge: <summary>"
git push origin master
```

推送 `master` 后，必须查看 GitHub Actions。通过后再同步线上：

```bash
ssh tx "cd /opt/ai-website && docker compose pull && docker compose up -d --force-recreate && docker compose ps"
```

部署后至少检查：

- `https://aiweb.bt199.com/api/health`
- 相关前台或后台页面
- 容器健康状态

如果只修改 Markdown 文档，GitHub Actions 可能不会触发，线上运行效果也不会变化；仍然需要推送 GitHub。

## 开发原则

修改前先理解现有实现，不要凭印象重写。

优先遵守现有代码风格：

- CommonJS 模块。
- Express 路由。
- EJS 模板。
- Node 内置 `node:test`。
- 不引入大型新框架，除非任务明确需要。

改动要小而完整：

- 修 Bug 时先补测试或复现用例。
- 新增行为要有针对性测试。
- 不做无关重构。
- 不格式化无关文件。
- 不回滚用户或其它 Agent 的改动。

## AI Agent 业务规则

栏目是网站建立后的稳定结构。已有栏目不能被 AI 大幅重命名、删除、合并或整体替换。AI 最多只能在冷启动时创建初始栏目，后续内容必须归入现有最匹配栏目。

文章必须有有效栏目。任何新建、更新、删除栏目逻辑都要保证文章不会变成无栏目或孤儿栏目。

文章生成要优先解决这些问题：

- 明显 AI 味。
- 公式化标题。
- 空泛摘要。
- 缺少事实锚点。
- 小标题像报告模板。
- 段落节奏过度均匀。
- 引用过时或无法核查的信息。

搜索和资料采集必须有兜底。Tavily 不可用时，仍要使用 DuckDuckGo、Bing、SearXNG、Bing News RSS、Google News RSS 或已管理 RSS 源继续工作。

AI Provider 失败时，程序应自动重试、降权、冷却或切换 Provider。不要因为一次失败就让整个自治循环停摆。认证错误可以禁用 Provider；网络、限流、服务端错误应优先重试和切换。

## 内容质量原则

网站明确标注内容由 AI 生成，但这不能成为低质量内容的借口。文章应尽量像真正编辑愿意发布的内容：

- 标题具体、自然，不像 SEO 题库。
- 开头直接进入事实、判断或场景。
- 每 300-500 字至少有一个具体事实锚点。
- 允许短段、转折和不完全对称结构。
- 少写“本文将”“深入分析”“挑战与机遇”“总的来说”等模板句。
- 不为显得具体而编造公司、日期、数字、报告、比例或案例。
- 资料不足时宁可缩小角度，也不要扩写空话。

质量门槛未通过的文章不能发布到前台、RSS 或 sitemap。

## 后台与前台 UI 原则

前台是内容站，不是营销落地页。设计应现代、干净、阅读优先。

- 首页要像资讯站，有清晰信息层级，不要杂乱堆卡片。
- 文章页正文要尽快出现，避免顶部无效留白过大。
- 目录不能遮挡正文。
- 移动端不能横向溢出。
- 文本不能被背景图或装饰元素覆盖。

后台是运营工具，应清晰、紧凑、可扫描。

- 表格长字段必须可换行或滚动，不能撑破页面。
- 状态显示必须来自真实日志或当前状态，不要长期错误显示“空闲”。
- 测试、保存等操作不应造成不必要跳转。
- 深色背景下输入框、按钮、编辑文字必须有足够对比度。

## 测试要求

常规修改至少运行：

```bash
npm test
```

涉及依赖、安全或上线前，运行：

```bash
npm audit --omit=dev
```

涉及语法敏感文件，可补充：

```bash
node --check server.js
node --check routes/admin.js
```

测试应覆盖实际风险，而不是只测实现细节。优先覆盖：

- AI Provider 选择和失败恢复。
- JSON 解析与修复。
- 文章状态流转。
- 栏目完整性。
- 分页。
- 后台关键 UI 行为。
- 质量检测和发布门槛。

## 数据与安全

不要提交真实密钥、线上数据库、日志或 `.env`。

不要对线上执行破坏性命令。涉及线上数据变更前，先备份：

```bash
ssh tx "cd /opt/ai-website && cp data/db.json data/db.json.bak-$(date +%Y%m%d-%H%M%S)"
```

后台密码、AI API Key、Provider 配置都属于敏感信息。输出日志和总结时不要泄露完整密钥。

## 交付标准

一次任务完成前，必须确认：

- 代码或文档已提交。
- 相关测试已通过，或明确说明为什么无法运行。
- 需要上线的改动已推送 GitHub 并同步线上。
- 本地开发服务器已关闭。
- 最终回复说明改了什么、验证了什么、是否已上线。

优先把事情做完，而不是只给建议。遇到阻塞时，先自行排查；确实需要用户决策时，再简短说明原因。
